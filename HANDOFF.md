# GVIDON TOUR — инструкция по сайту и админ-панели

## Часть 1. Для владельца сайта

### Вход в админ-панель

Адрес: `https://<ваш-домен>/admin/login` (сейчас — `https://gvidon-tour.vercel.app/admin/login`).

Логин и пароль вам передаёт разработчик отдельно (не через email/мессенджер в открытом виде, а лично или через защищённый канал). Если пароль забыт — это может сбросить только разработчик (через технический скрипт, доступа "забыл пароль" на сайте нет).

### Что можно делать в панели

- **Обзор** — сводка: сколько туров, фото, отзывов, статей опубликовано.
- **Заявки** — все заявки с сайта (имя, телефон, интересующий тур). Можно отмечать как "обработано" и удалять. При новой заявке приходит уведомление в Telegram.
- **Медиа** — общая библиотека фото. Загрузите сюда фото один раз — потом вставляйте ссылку на него в любой раздел (тур, галерея, отзыв, статья, достижение).
- **Туры** — карточки туров/экскурсий: название, описание, цена, программа по дням на трёх языках (рус/англ/каз). Переключатели "опубликован" и "рекомендуемый" (звёздочка на главной).
- **Галерея** — фото и видео для страницы "Галерея".
- **Отзывы** — отзывы клиентов с рейтингом.
- **Блог** — статьи.
- **Достижения** — плашки с наградами/цифрами на главной странице.

Все изменения появляются на сайте сразу после сохранения — обновлять ничего дополнительно не нужно.

### Чего лучше не делать

- Не публикуйте тур/статью, пока не заполнены все три языка (RU/EN/KK) — иначе на английской/казахской версии сайта будет пусто.
- Не удаляйте фото из "Медиа", если оно ещё используется в туре, отзыве или статье — там просто пропадёт картинка.

## Часть 2. Для того, кто технически сопровождает сайт

### Хостинг

- Сайт и админка — Vercel, деплой автоматически при `git push` в ветку `main`.
- База данных и файловое хранилище — Supabase (Postgres + Storage).
- Уведомления о заявках — Telegram Bot API.

### Переменные окружения (заданы в Vercel → Settings → Environment Variables и локально в `.env`)

`DATABASE_URL`, `DIRECT_URL`, `AUTH_SECRET`, `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`, `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`.

**Важно:** в процессе разработки значения `SUPABASE_SERVICE_KEY`, `TELEGRAM_BOT_TOKEN` и пароль базы данных передавались через переписку с AI-ассистентом (Claude). Такие секреты стоит считать потенциально попавшими в чужие логи и **сменить перед долгосрочной эксплуатацией**:

1. **Supabase service key** — Supabase Dashboard → Project Settings → API → Regenerate `service_role` key. После смены обновить `SUPABASE_SERVICE_KEY` в `.env` и в Vercel, передеплоить.
2. **Пароль базы данных** — Supabase Dashboard → Project Settings → Database → Reset database password. После смены обновить пароль внутри `DATABASE_URL` и `DIRECT_URL` и в `.env`, и в Vercel, передеплоить.
3. **Telegram bot token** — у @BotFather команда `/mybots` → выбрать бота → `API Token` → `Revoke current token`. Обновить `TELEGRAM_BOT_TOKEN` в `.env` и в Vercel, передеплоить.
4. `AUTH_SECRET` уже был сгенерирован заново на этапе передачи (2026-07-09) — можно не трогать, но при желании его тоже можно сменить в любой момент (это просто разлогинит текущие сессии админки).

### Пароль администратора

Финальный пароль для входа в `/admin` также был сгенерирован на этапе передачи и передан владельцу лично (не хранится в репозитории). Сменить пароль или создать нового администратора:

```bash
npx tsx scripts/create-admin.ts <логин> <пароль> [имя]
```

### Резервные копии

```bash
npm run db:backup   # сохраняет все таблицы в backups/backup-<дата>.json (папка не попадает в git)
npm run db:restore -- backups/backup-2026-07-09....json   # восстанавливает записи по id (добавляет/перезаписывает, ничего не удаляет)
```

Рекомендуется делать `db:backup` вручную перед крупными правками и хранить файлы отдельно от сервера (например, в облачном хранилище), так как папка `backups/` не коммитится в git и не резервируется автоматически.

### ⚠️ Опасная команда: `npm run db:seed`

Это скрипт первоначального наполнения базы демо-контентом. Он **защищён**: если в базе уже есть данные, скрипт откажется работать и ничего не сотрёт. Явно обойти защиту (например, для чистой базы на новом сервере) можно так:

```bash
SEED_FORCE=1 npm run db:seed          # bash
$env:SEED_FORCE=1; npm run db:seed    # PowerShell
```

Без крайней необходимости — не запускать на боевой базе.

### Безопасность (аудит 2026-07-09)

- Соединение с Postgres принудительно шифруется на уровне кода (`src/lib/db.ts` добавляет `sslmode=require&uselibpqcompat=true`). Рекомендуется дополнительно включить в Supabase Dashboard → Project Settings → Database → **«Enforce SSL on incoming connections»** — тогда нешифрованные подключения будет отвергать сам сервер.
- Принудительный выход всех сессий админки (например, при подозрении на утечку): сменить `AUTH_SECRET` в Vercel и передеплоить. Секрет генерируется командой `node -e "console.log(require('crypto').randomBytes(32).toString('base64url'))"` — лучше сгенерировать его самостоятельно и никому не пересылать.
- В Vercel Dashboard → Firewall при DDoS-атаке можно включить **Attack Challenge Mode** (проверка браузера для всех посетителей).
- Файлы бэкапов (`backups/*.json`) содержат заявки клиентов (имена/телефоны/email) и хэш пароля админа — хранить их как конфиденциальные.

### Перенос на свой сервер (VPS) в будущем

Приложение не завязано на Supabase Auth (своя JWT-авторизация) — переезд требует только:
- обычный Postgres вместо Supabase Postgres (те же таблицы через `prisma db push`);
- замены модуля `src/lib/storage.ts` на работу с диском/MinIO вместо Supabase Storage — это единственное место в коде, которое обращается к Supabase Storage API.
